Benchmark 与论文/补充材料一致性检查

本报告只检查 benchmark 层面的实验规模、setup、数据生成脚本和已有结果文件是否能支撑论文与补充材料中的表述。 结论是：部分结果数值可以对上论文表格，但当前仓库的若干 benchmark 脚本、数据规模和实验语义已经与论文版本漂移。

检查范围：benchmark / results / data / scripts / Java runner 论文文件：ISWC26_ProbSPARQL.pdf 补充材料：ISWC_SupplementaryMaterial.pdf 检查日期：2026-05-10
高风险 6 项 会影响论文表格的可复现性或实验语义解释。
中风险 3 项 数值可对上，但路径、命名或版本信息不干净。
基本一致 2 项 运行轮次和 filter-pushdown 数值基本对齐。

总体结论

需要对齐

如果目标是“代码和 benchmark artifact 向论文对齐”，优先级应放在 Exp3 divergence-decision、Table S.6 end-to-end DIVJOIN、 dimensionality scaling 和 Exp4 datatype extensibility。它们不是简单的文档命名问题，而是当前数据规模、生成逻辑、输出指标或结果文件结构与论文声明不同。

论文已经固定的前提下，建议不要再用“当前实现新增需求后的实验”去解释论文表格；应该保留或重建一套 paper-aligned benchmark runner/data/results。

Exp3 Divergence Decision

高风险
维度 论文/补充材料声明 当前仓库状态 影响
workload 规模 Easy / Medium / Hard / Mixed 每个 workload 都是 N=2400 candidate pairs。 benchmark/data/exp3 当前每个 workload 是 300 pairs；每个 TTL 有 600 个 prob:hasGMM triple，ground truth 总共 1200 行数据。 当前数据不能支撑论文 Table S.5 / S.6 的 N=2400 声明。
reference JSD 补充材料写 reference JSD 使用 10^6 Monte Carlo samples。 benchmark/scripts/Experiments3/generate_exp3.py 中生成 pair 时使用 50000 samples。 accuracy / MAE 的 reference provenance 不一致。
difficulty 分桶 Easy: m >= 0.15；Medium: 0.05 <= m < 0.15；Hard: m < 0.05。 当前脚本写 Easy: |JSD-0.3| > 0.2；Medium: 0.05 < |JSD-0.3| <= 0.2；Hard: <= 0.05。 Easy/Medium 的难度区间和论文不是同一批 workload 定义。
默认数据生成参数 论文表格隐含使用固定 paper workload。 脚本文档写旧官方数据是 K=5, N=300，但 CLI 默认值是 --n 100 --k 2。 重新生成数据时很容易生成出与论文完全不同的 workload。
结果表 MAE 论文 Table S.5 中 SPRT 与 Det-Bound 有各自 MAE。 当前 benchmark/results/exp3/exp3_classification.csv 中 V3_SPRT 和 V4_BOUNDS 的 F1/latency 与论文一致，但 MAE 看起来互换。 即使数值接近论文，当前 CSV 的 method-MAE 对应关系也不可靠。

Table S.6 End-to-End DIVJOIN

高风险

补充材料 Table S.6 报告的是 Easy / Medium / Hard / Mixed 四个 workload 下，五种 divergence-decision strategy 的完整 DIVJOIN query runtime。 当前仓库没有发现可以直接对应 Table S.6 的结果文件。

  • benchmark/results/exp2/legacy_5variants 是 mixed-K / unimodalFrac / selectivity 实验，列是 NPairs、UnimodalFrac、Selectivity、Theta，不是 Easy/Medium/Hard/Mixed。
  • src/main/java/org/apache/jena/probsparql/Exp2Benchmark.java 当前语义是 InEngine_CheapFirst / InEngine_JSDFirst / DIVJOIN 三路比较，不是论文 Table S.6 的五策略完整 DIVJOIN 表。
  • 因此 Table S.6 目前属于“论文有结果，但当前 benchmark 目录没有 paper-aligned runner/result”的状态。

Dimensionality Scaling

高风险

补充材料 Table S.7 的实验固定 K=3，维度是 d={1,2,4,8}，使用固定 S3 graph。 当前仓库的 dimension benchmark 已经改成另一套维度和结果语义。

  • benchmark/scripts/Experiments1/dimension/run_exp1_dimension.sh 默认 DIMENSIONS="1 3 5 10"。
  • benchmark/scripts/Experiments1/dimension/generate_exp1_dimension.py 默认 DEFAULT_DIMENSIONS = [1, 3, 5, 10]。
  • benchmark/results/exp1/dimension/exp1_dimension_summary.csv 也对应 d=1/3/5/10，并且同时报告 prob:jsdivergence 和 prob:jsd，不是论文 Table S.7 的 d=1/2/4/8 表。

Exp4 Datatype Extensibility

高风险

论文补充材料 S.8.9 声称在 controlled 500-entity subset 上比较 GMM、Dirichlet、Histogram，并报告 median per-comparison latency； cross-type Table S.8 报告 Precision / Recall / F1 / Latency。当前 Exp4 benchmark 的输出结构并不对应这张表。

项目 论文声明 当前仓库状态
same-type datatype latency controlled 500-entity subset，per-comparison latency: histogram 0.1ms、Dirichlet 1.2ms、GMM 3.8ms。 Exp4MicroBenchmark 使用 N=1000，输出 per-call microseconds，pairwise JSD 查询是 LIMIT 200。
cross-type evaluation epsilon=0.1，10^6-sample references，输出 Precision / Recall / F1 / Latency。 Exp4CrossTypeJSD 使用 N_PAIRS=100、MC_SMALL=5000、MC_CROSS=10000，输出 SameTypeJSD / CrossTypeJSD / AbsError / TimeNs。

这一部分是目前对齐最好的 benchmark。补充材料 Table S.4 的 pass fraction 和 latency 数值可以在 benchmark/results/exp5/legacy_ab_labels/exp5_summary_all.csv 中找到。

  • pass fractions: 0.01、0.05、0.10、0.30。
  • in-engine / post-processing latency 和 speedup 与论文表格一致。
  • 轻微问题是结果路径带 legacy_ab_labels，而当前 run_exp5.sh 默认输出到 benchmark/results/exp5。

Execution Setup

中风险
  • 多数 runner 使用 3 warm-up 和 10 measured runs，和论文/补充材料一致。
  • 当前本机 Java 是 21，和论文一致。
  • pom.xml 中 jena.version 是 6.0.0-SNAPSHOT，而论文写 Jena 6.0.0；如果严格做 artifact reproducibility，这里需要说明或固定版本。
  • Exp2Benchmark.java 类注释仍写过时的 WARMUP=1 + RUNS=3，但实际代码和 shell runner 是 3/10。

建议对齐顺序

建议
  1. 先重建 paper-aligned Exp3：N=2400、K 固定、10^6 reference、difficulty margin 按补充材料定义，并重新生成 Table S.5。
  2. 补齐 Table S.6 对应的完整 DIVJOIN runner 和结果文件，不要用当前 Exp2 mixed-K benchmark 替代。
  3. 把 dimension scaling 的 paper runner 固定为 d={1,2,4,8}，与当前新增的 d={1,3,5,10} 实验分开。
  4. 重写或新增 Exp4 paper runner，使其输出论文 Table S.8 所需的 P/R/F1/Latency，而不是只输出 JSD error/time。
  5. 清理 S1-S4 与 E1-E7 的命名关系；如果使用 Git LFS，README 里必须明确需要拉取 LFS 文件后才能复现 graph-size 表。
